## Two workflows, one interlock (the seats are separate instances)

There is NO single workflow across both seats. There are TWO, coordinated
by the courier + the gate:
- **cc2's EXPLORATION workflow** (their seat): fan-out of design cells per
  loop → packet.
- **cc's VERIFICATION workflow** (my seat, this machine): re-derive each
  packet cell independently → gate → bank → chain the next loop.

## THE PER-LOOP MACHINE (my side; runs on each incoming packet)

phase 1 — INTAKE: unpack, privacy-scan, hash-match the packet.
phase 2 — ADVERSARIAL VERIFY (fan-out, parallel): each packet claim
  re-derived by an INDEPENDENT method (not cc2's) + a BASE-RATE control
  cell on every numeric match (the firewall, automated). A claim banks
  only if the independent re-derivation agrees AND the base-rate cell
  says low-base-rate. Disagreement → STOP-and-diagnose (never a silent pass).
phase 3 — FIREWALL GATE: any SM-facing quantity → Gate-5 fence; any
  small-smooth match → base-rate kill unless FORCED. Convention/input-
  sensitivity check on any "nice value" (the θ₀ lesson).
phase 4 — BANK: survivors → B#, locks, ledgers, atlas, PR, mirror.
phase 5 — CHAIN: the banked results seed the next loop's design (fed back
  to cc2 via the courier); loop-until-dry (CHARTER: two partial-only loops
  end it).

## THE ADVANCED FEATURES (why "advanced", not just a loop)

- **Adversarial verification by default**: every claim re-derived a second,
  independent way before banking (the two-route discipline that caught the
  support-test Nahm substitution).
- **Automated base-rate control**: every numeric coincidence gets a
  base-rate cell (the B615/B655/θ₀/7983360 pattern) — no match banks on
  proximity alone.
- **Convention/robustness probe**: any "clean value" is re-computed under
  ≥2 conventions/inputs; non-robust ⇒ firewalled (the θ₀=2/9 lesson).
- **Loop-until-dry**: loops continue until two return nothing new; the tail
  is not truncated silently.

## THE ONE SHARED GATE (the interlock)

halves that must agree EXACTLY before any verdict (Γ(5)/modular vs
holonomy/Nahm). This is the only place the two workflows synchronize;
everywhere else they run in parallel, joined by the courier.
